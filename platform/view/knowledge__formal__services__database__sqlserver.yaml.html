  - id: async-writes-return-request-id
    rule: "write calls answer with AsyncResponse {request_id, resource:{id}} (api_docs response examples, shared DBaaS envelope) — request_id is the ONLY key to GET /v1/requests/{request_id} (sqlservershowrequest); capture it from the create response, no list exposes it"
    provenance: docs
  - id: list-parameters-needs-group-id
    rule: "GET /v1/parameters REQUIRES query param dbaas_parameter_group_id (shared DBaaS DTO) — harvest it from GET /v1/parameter-groups contents[].id first"
    provenance: docs
  - id: engine-editions-2019-2022
    rule: "engine catalog = SQL Server 2019 (Enterprise/Standard, EOS 2029-06, EoTS 2030-01) and SQL Server 2022 (Enterprise/Standard, EOS 2032-06, EoTS 2033-01) (userguide overview). Edition is baked into the engine-version id; minor-only upgrade WITHIN a major (no 2019→2022) — '동일한 Major 버전 내에서 Minor 버전 업그레이드 기능만 지원합니다' (managing)"
    provenance: docs
  - id: secondary-needs-license-and-enterprise
    rule: "Secondary (Always On HA standby / read-only replica) REQUIRES a valid SQL Server License Key input + Enterprise Edition + backup enabled, and ONLY ONE per cluster — '하나의 클러스터당 1개만 구성 가능', '입력한 라이선스 key가 유효하지 않을 경우 서비스가 생성되지 않을 수 있음' (secondary how_to). The license key is an account/vendor credential with NO documented self-service issuance — ESCALATE (credential gate)"
    provenance: docs
  - id: additional-databases-enterprise-100
    rule: "Enterprise edition supports up to 100 additional databases per cluster, added/removed via POST /v1/clusters/{id}/databases {add_databases, del_databases} (managing). Standard edition limit not documented"
    provenance: docs
  - id: secondary-name-rule
    rule: "Secondary name = 3–15 chars, lowercase-start, lowercase/digits/hyphen — '영문 소문자로 시작하여, 소문자, 숫자와 특수문자(-)를 사용하여 3 ~ 15자로 입력' (secondary how_to). NOTE: differs from relational replica name pattern (3–20)"
    provenance: docs
  - id: server-type-naming
    rule: "server-type name = [Type][Spec][vCore][Memory], e.g. 'Standard db1v2m4' = 2 vCore / 4GB; Type=Standard|High Capacity, Spec=db1 (standard) or dbh2 (large, 24+ vCore) (userguide overview) — consume the NAME from GET /v1/server-types contents[].name, not an id"
    provenance: docs
  - id: block-storage-bounds
    rule: "DATA storage capacity 16–5,120 GB in 8 GB increments; restore requires the same storage type as origin (backupandrestore how_to)"
    provenance: docs
  - id: default-db-port-2866
    rule: "Database Port defaults to 2866 but is user-overridable — 'Database Port는 기본으로 2866이나, 사용자가 지정하셔도 됩니다' (connection how_to)"
    provenance: docs

quirks:
  - id: license-excludes-from-shared-heavy
    note: "SQL Server is the only DBaaS engine excluded from heavy-shared-dbaas: cluster create itself does NOT take a license field (instance create can be tested without one), but Secondary/Always On HA does. The guarded source sends empty ids so create can only 4xx; this model curates real tokens but the cluster create template has NEVER returned 2xx → docs/UNPROVEN"
    provenance: docs
  - id: secondary-is-not-relational-replica
    note: "sqlserver has NO /v1/clusters/{id}/replicas, promote, reset-replica, sync-replica-state, other-region or archive endpoints (unlike postgresql/mysql). Its HA/read-scale path is add-secondary (POST /v1/clusters/{id}/add-secondary {license, name, service_ip_address}) + databases (POST /v1/clusters/{id}/databases). switchover applies to the Always On Active↔Standby pair once a Secondary exists"
    provenance: docs
  - id: backup-scheduled-not-immediate
    note: "backup runs on schedule not immediately — '백업을 설정한 경우에는 설정한 시간 이후 지정된 시간에 백업이 수행되며'; restore (PITR '사용자 지정 시점' or backup point) needs a COMPLETED backup first, PITR window = first backup start .. now-minus-archive-granularity (5/10/30min or 1hr). Restore creates a NEW cluster ('신규 서버가 생성되고') inheriting origin network. So create-then-immediately-restore fails for lack of a backup (backupandrestore how_to)"
    provenance: docs
